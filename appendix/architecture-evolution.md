# 库表容量增长与架构决策：第一性原理学习地图

> 适合谁看：正在经历数据增长的工程师、准备做架构选型的 Tech Lead、想建立架构师视角的开发者。
> 不适合谁看：想直接抄分库分表方案的（本文不会给你具体脚本）。

这份内容的目标不是教你「分库分表怎么做」，而是建立一个**架构师视角**：

```text
业务增长
    ↓
数据增长
    ↓
资源瓶颈
    ↓
架构演化
    ↓
新问题出现
    ↓
继续演化
```

理解这条主线后，你会发现：

```text
分库分表不是目标
容量问题也不是问题
系统演化能力才是问题
```

---

## 一、先问一个根本问题

### 为什么库表容量增长会成为问题？

很多人第一反应：

```text
数据太多了
```

其实不准确。数据库不会因为数据多而崩溃。

真正的问题是：

```text
资源有限
```

数据库本质：

```text
CPU
↓
Memory(Buffer Pool)
↓
Disk(Page)
↓
Network
```

当数据增长时：

```text
B+Tree变大
↓
Page增多
↓
缓存命中率下降
↓
磁盘IO增加
↓
响应时间变长
```

因此：

```text
容量问题
=
资源问题
```

---

## 二、容量增长的三个阶段

### 第一阶段：业务初创期

典型规模：

```text
用户：1万
订单：10万
数据：GB级
```

架构：

```text
Nginx
    ↓
应用
    ↓
MySQL
```

核心问题：

```text
业务验证
```

不是：

```text
高并发
```

正确策略：

```text
单库单表
```

错误策略：

```text
微服务
Kafka
ES
分库分表
```

---

### 第二阶段：业务成长期

典型规模：

```text
用户：100万
订单：千万级
数据：百GB
```

出现：

```text
慢SQL
CPU升高
Buffer Pool不足
磁盘IO升高
```

此时：

```text
问题还在单机范围
```

正确顺序：

```text
SQL优化
↓
索引优化
↓
缓存
↓
增加内存
↓
读写分离
```

不要立即：

```text
分库分表
```

---

### 第二阶段补充：什么时候真的需要分库分表？

以下**量化指标**比直觉更可靠：

| 指标 | 经验阈值 | 说明 |
|------|---------|------|
| 单表行数 | > 2000万 | InnoDB 经验值，B+Tree 层级开始影响性能 |
| 单表大小 | > 20GB | 影响备份、DDL、Buffer Pool 命中率 |
| 写入 TPS | > 5000 且已优化到极限 | 单实例写入达到天花板 |
| 磁盘容量 | > 单机上限的 70% | 预留 30% 缓冲 |

> 注意：这些是**参考值**，不是绝对值。如果 SQL 已经优化到极致，缓存命中率仍持续下降，才是真正到了需要水平扩展的时刻。

---

### 第三阶段：业务成熟期

典型规模：

```text
用户：千万
订单：数十亿
数据：TB级
```

出现：

```text
单机容量达到极限
单机写入达到极限
```

这时：

```text
必须水平扩展
```

进入：

```text
分区
↓
分库分表
↓
分布式数据库
```

## 三、数据库容量问题分类

很多团队混为一谈。实际上完全不同。

### 类型1：存储容量问题

现象：

```text
磁盘快满了
```

例如：5TB、10TB、20TB。

解决方案：

#### 归档

历史数据迁移：

```sql
-- order → order_history
INSERT INTO order_history SELECT * FROM `order` WHERE create_time < '2023-01-01';
DELETE FROM `order` WHERE create_time < '2023-01-01'; -- 分批删
```

#### 冷热分离

```text
近一年数据 → 热数据 → 主库
历史数据   → 冷数据 → 归档库
```

#### 扩容磁盘

最简单直接。

---

### 类型2：查询性能问题

现象：

```text
SQL越来越慢
```

根因：

```text
索引失效
回表过多
排序
聚合
```

解决：

```text
EXPLAIN 分析
```

优化方向：

```text
覆盖索引
联合索引
分页优化
```

收益通常：

```text
10倍以上
```

---

### 类型3：写入性能问题

现象：

```text
TPS持续升高
```

例如：1000TPS → 10000TPS → 100000TPS

出现：

```text
锁竞争
日志压力
CPU瓶颈
```

此时：

```text
单机扩容无效
```

需要：

```text
水平扩展
```

---

## 四、架构演化路线对比

### 方案1：单库单表

**优点**：简单、事务完整、Join 简单、维护成本低

**缺点**：容量有限

**适用**：绝大多数创业项目

---

### 方案2：缓存（Redis）

**解决**：读压力

架构：

```text
App
 │
 ├── Redis
 │
 └── MySQL
```

**新问题**：缓存一致性

**收益**：极高

**复杂度**：低

**推荐指数**：★★★★★

---

### 方案3：读写分离

**解决**：查询压力

架构：

```text
Master
   │
   ├── Slave1
   │
   └── Slave2
```

**新问题**：复制延迟

表现：

```text
刚写入，查询不到
```

**推荐指数**：★★★★☆

---

### 方案4：分区表

例如：

```sql
CREATE TABLE `order` (
    id BIGINT NOT NULL,
    create_time DATETIME NOT NULL,
    ...
    PRIMARY KEY (id, create_time)
) PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2021 VALUES LESS THAN (2022),
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

**优势**：

```text
逻辑一张表
物理多分区
```

**解决**：

```text
时间维度数据增长
```

典型场景：

```text
订单表
日志表
```

**关键限制**：

```text
MySQL 5.7：单表最多 1024 个分区
MySQL 8.0：单表最多 8192 个分区

更重要的是：
分区键必须是主键或唯一键的一部分
↓
这可能导致你无法按想要的方式分区
```

**推荐指数**：★★★★★

> 很多公司可以用分区解决问题，而不是分库分表。

---

### 方案5：分库分表

**解决**：

```text
单机容量
单机写入
```

例如：

```text
order_0000
order_0001
order_0002
...
order_1023
```

**新问题**：

```text
跨库事务
跨库 Join
跨库聚合
数据迁移
```

**复杂度**：极高

**推荐原则**：

```text
最后使用
```

---

### 方案6：分布式数据库

对应产品：TiDB、OceanBase、CockroachDB

**解决**：

```text
超大规模扩展
```

**新问题**：

```text
运维复杂
学习成本
SQL兼容性
```

**适用**：

```text
大型企业
```

---

## 五、架构升级引入的新问题

这是最重要的一张表：

| 升级方案 | 解决的问题 | 引入的新问题 |
|---------|-----------|------------|
| Redis | 数据库读压力 | 缓存一致性 |
| 主从复制 | 读扩展 | 复制延迟 |
| 分区表 | 大表扫描 | 分区设计 & 主键约束 |
| 分库分表 | 容量与写入 | 跨库事务/Join/聚合/迁移 |
| 微服务 | 团队协作 | 分布式复杂度 |
| 分布式数据库 | 水平扩展 | 运维复杂度 & SQL 兼容性 |

核心规律：

```text
没有免费的架构升级
```

任何升级都是：

```text
解决旧问题
+
引入新问题
```

### 补充：回退成本

这是很多架构决策忽略的维度：

```text
升级容易，降级难。

一旦上了分库分表，想回到单库单表：
    几乎不可能。

一旦用了某个分布式数据库：
    迁移成本极高。

因此架构升级的另一个原则是：
    尽可能选择可逆的决策。
```

| 方案 | 可逆性 | 回退成本 |
|------|-------|---------|
| SQL优化 | ✅ 完全可逆 | 零 |
| 索引优化 | ✅ 可逆（删索引） | 低 |
| Redis缓存 | ✅ 可逆（去掉缓存层） | 低 |
| 读写分离 | ✅ 可逆 | 中 |
| 分区表 | ⚠️ 部分可逆 | 中 |
| 分库分表 | ❌ 几乎不可逆 | 极高 |
| 分布式数据库 | ❌ 不可逆 | 极高 |

## 六、架构决策框架

每次升级前问四个问题：

### 问题1：当前瓶颈是什么？

```text
CPU？
内存？
磁盘？
网络？
```

用监控数据回答，不要凭感觉。

### 问题2：是否有更简单方案？

例如：

```text
SQL优化
索引优化
缓存
```

把优化做到极致后，再考虑架构变更。

### 问题3：升级后新增什么复杂度？

例如：

```text
缓存一致性
跨库事务
复制延迟
```

### 问题4：收益是否大于复杂度？

如果回答不了：

```text
不要升级
```

## 七、一页纸架构演化图

```text
单库单表
    │
    ├── SQL优化 ← 始终要做
    │
    ├── 索引优化 ← 始终要做
    │
    ▼
缓存层（Redis）
    │
    ▼
读写分离
    │
    ├── 分区表 ← 如果是时间维度增长
    │
    ▼
分库分表 / 分布式数据库
```

> 实际架构决策往往不是严格的线性递进。SQL优化和索引优化应该**持续进行**；
> 缓存可以在任何阶段引入；分区表适用于时间维度增长，分库分表适用于写入扩展——它们是解决不同问题的平行方案。

## 八、常见误区

### 误区1：分库分表能解决所有性能问题

```text
事实：如果是 SQL 写法问题，分了库也一样慢。
先查 EXPLAIN，再优化 SQL，最后才考虑架构变更。
```

### 误区2：分库分表后性能一定更好

```text
事实：跨库查询反而更慢。
原来一条 SQL 搞定的事，现在需要应用层聚合。
```

### 误区3：先分库分表，以后就不用管了

```text
事实：数据分布不均、热点问题会不断出现。
分库分表是一个持续运维的过程，不是一次性的操作。
```

### 误区4：分库分表是数据库层面的事

```text
事实：它深刻影响应用层代码。
路由逻辑、跨库聚合、分布式事务——都写在你业务代码里。
```

### 误区5：数据量大了就应该分区 / 分表

```text
事实：先问「瓶颈是什么」。
如果磁盘还没满、查询还很快，分区只能增加复杂度。
```

### 误区6：架构演化是线性的

```text
事实：不同路径可以并行。
你可以同时有 Redis 缓存 + 分区表 + 读写分离，
不一定非要先走完一条路再走另一条。
```

---

## 最终结论

很多工程师思考的是：

```text
什么时候分库分表？
```

而架构师思考的是：

```text
当前瓶颈是什么？
最小代价的解决方案是什么？
未来还能演化吗？
```

因此关于库表容量增长，真正的第一性原理公式是：

```text
业务增长
    ↓
资源瓶颈
    ↓
定位约束资源
    ↓
最小复杂度优化
    ↓
持续反馈
    ↓
系统演化
```

这比记忆各种中间件、分库分表框架和数据库产品更重要，因为它适用于**任何规模的系统**。
