# 第六章：MVCC

> 索引解决「如何快速找到数据」，事务解决「如何保证数据正确」，MVCC 解决「如何既正确又高并发」。这是 InnoDB 最精妙的设计之一。

---

## 6.1 MVCC 到底在解决什么？

### 第一性原理

数据库有两个目标：

```text
目标1：数据正确
目标2：速度快
```

这两个目标天然矛盾。

### 并发读写问题

```sql
-- 事务A
UPDATE account SET balance=900 WHERE id=1;

-- 事务B
SELECT balance FROM account WHERE id=1;
```

B 看到什么？1000 还是 900？—— 这就是**并发读写问题**。

### 最简单方案：加锁

A 修改时加锁 → B 查询时等待 → A 提交后释放锁 → B 查询。

数据正确。但 10000 个查询全部排队 → **数据库性能崩溃**。

---

## 6.2 版本化思想

重新思考：为什么 B 必须等？B 只是查询，真的必须看到 900 吗？看到 1000 也可以，因为 **1000 是真实存在过的数据**。

于是出现革命性思想：**不要覆盖数据，而是保存历史版本**。

```text
Version1: balance=1000
Version2: balance=900  ← 事务A修改

查询（事务B）：读 1000
写（事务A）：写 900
互不影响。
```

这就是 **MVCC（Multi-Version Concurrency Control，多版本并发控制）**。

---

## 6.3 MVCC 本质

一句话概括：**读历史版本，写最新版本**。

```text
Read 不阻塞 Write
Write 不阻塞 Read
```

达到**高并发**。

---

## 6.4 Undo Log 的新身份

上一章学过 Undo Log 用于回滚 — 这是第一层用途。

第二层用途：**保存历史版本**。

```text
原数据：balance=1000
事务A执行 UPDATE →
  Undo Log 保存：1000
  数据页更新为：900

当前版本：900
  │
  ▼
Undo：1000

再次更新为 800：
800 → 900 → 1000
形成 版本链
```

---

## 6.5 隐藏字段

你以为表里只有 `id` 和 `name`？实际上 InnoDB 偷偷加了字段：

```text
id
name
trx_id      ← 哪个事务修改的
roll_pointer ← 指向上一个版本
```

- **trx_id**：事务 100 写入 balance=1000；事务 101 修改后 balance=900, trx_id=101
- **roll_pointer**：像链表一样，指向旧版本

---

## 6.6 Read View

关键：数据库怎么决定读哪个版本？

答案：**Read View**。可以理解为**拍照**。

事务开始时拍一张快照，记录此刻有哪些事务存在。以后查询都按这张照片看。

### 举例

时刻 T1：余额 1000。事务 A 开始，Read View（快照建立），看到 1000。

时刻 T2：事务 B 执行 `UPDATE balance=900` 并提交。数据库真实值：900。

事务 A 再次查询 → **看到 1000**。

因为：**事务 A 活在自己的快照里**。这就是 **Repeatable Read（可重复读）**。

---

## 6.7 为什么 Repeatable Read 成立？

因为 **Read View + Undo Log** 组合：

```text
查询 → 发现当前版本太新 → 沿 Undo 链回退 → 找到符合快照的版本 → 返回
```

---

## 6.8 隔离级别

数据库面临一个选择：到底允许用户看到什么？SQL 标准定义了 4 种隔离级别：

### Read Uncommitted（最低）

事务 A 没提交的 900，事务 B 直接看到。A 回滚了 → B 看到了不存在的数据 → **脏读**。

### Read Committed（Oracle 默认）

只看已提交数据。每次查询重新拍照。第一次 1000，第二次 900 → 不一致。

### Repeatable Read（MySQL 默认）

第一次 1000，第二次仍然 1000。因为同一张快照。

### Serializable（最高）

全部加锁，像单线程执行。性能最差。

---

## 6.9 MVCC 为什么这么强？

- 传统数据库：读等写、写等读
- MVCC：读旧版本、写新版本 → **读写分离**

高并发场景：10000 个查询同时执行，几乎无需等待。

---

## 6.10 MVCC 的边界

MVCC 只能解决**读写冲突**，不能解决**写写冲突**。

事务 A 减 100，事务 B 减 50，同时执行 → 必须决定谁先写，否则数据错误。

这时：**Lock 登场**。

---

## 完整链路

```text
事务
│
├── ACID
├── Redo Log → 保证提交不丢
├── Undo Log → 保存历史版本
├── Version Chain
├── Read View
├── MVCC → 读历史版本
└── Repeatable Read
```
