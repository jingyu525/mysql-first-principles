# 第五章：事务与 ACID

> 索引解决「如何快速找到数据」，事务解决「如何保证数据永远正确」。这是从性能问题进入正确性问题的转折点。

---

## 5.1 为什么需要事务？

### 假设：一条 SQL 一定是原子的吗？

```sql
UPDATE account SET balance = balance - 100 WHERE id = 1;
```

很多初学者认为：执行 → 成功。但实际上 MySQL 最终会拆成很多步骤：

```text
找到记录 → 读取 Page → 修改内存 → 更新索引 → 写日志 → 写回磁盘
```

实际上是**多个动作**，不是一个动作。

### 转账案例

```text
A = 1000, B = 1000
A - 100, B + 100
```

逻辑上是"转账"一件事，实际上是两件事。如果 A 扣钱成功 → 机器断电 → A=900, B=1000，**钱消失了**。

### 第一性原理

事务的本质：**让多个操作看起来像一个操作**。

**All or Nothing**：要么全部成功，要么全部失败。

---

## 5.2 事务是什么？

事务可以理解为一个**状态机**：

```text
          Begin
             │
             ▼
       Running
         /   \
        /     \
 Commit       Rollback
```

数据库永远不允许"执行一半结束"这种状态暴露给用户。

---

## 5.3 ACID 逐一推导

很多人背 ACID 然后忘了。实际上 ACID 是四个问题：

### A：Atomicity（原子性）

**问题**：做了一半怎么办？

**解决**：全部成功 or 全部失败。

转账：`A-100` 和 `B+100` 必须一起成功。

### C：Consistency（一致性）

**问题**：数据规则会不会被破坏？

数据库里有很多**规则**（约束），一致性要求事务结束后，所有规则依然成立。

#### 规则 1：总金额不变的例子

转账前：A=1000, B=1000，总金额 2000。

```text
-- 事务执行中
A = A - 100  →  A = 900
-- ⚡ 此时断电！A=900, B=1000，总金额 1900
```

如果此时结束，规则被破坏：**钱消失了 100 块**。

一致性要求：要么全部完成（A=900, B=1100，总额 2000），要么全部回滚（A=1000, B=1000，总额 2000）。

**关键理解**：执行中途的状态（总额 1900）绝不能对外暴露。

#### 规则 2：余额不能为负

```sql
CREATE TABLE account (
    id INT PRIMARY KEY,
    balance DECIMAL(10,2) CHECK (balance >= 0)
);
```

如果 `A.balance = 50`，你执行 `A - 100`，结果是 `-50`，违反 CHECK 约束。事务必须失败。

#### 规则 3：外键约束

```text
用户表：user_id=1  ←  订单表：order_101(user_id=1)
```

删除 `user_id=1` 时，订单表中的引用怎么办？外键约束要求：要么连订单一起删，要么禁止删除用户。

#### 规则 4：主键唯一

```text
INSERT INTO t VALUES (1, 'Alice');  -- OK
INSERT INTO t VALUES (1, 'Bob');    -- 违反！同一个主键重复
```

**总结**：一致性 = 事务开始前数据库是合法的 → 事务结束后还是合法的。中间状态无论多乱，都不能被外界看到。这四条规则只是一个引子——真实数据库还有更多约束类型（UNIQUE、NOT NULL、触发器、级联更新等）。

> **提示**：一致性（C）和其他三个（AID）的关系很微妙。有人说"有了 AID，自然就有 C"，因为原子性保证 All or Nothing，隔离性保证并发不出错，持久性保证提交后不丢数据——三者加起来，数据库自然就能维护一致性。所以你主要关注 AID 即可，C 是它们的**结果**。

### I：Isolation（隔离性）

**问题**：多个事务同时执行怎么办？

事务 A 修改余额，事务 B 查询余额。B 应该看到旧值还是新值？这就是隔离问题，后面 MVCC 专门解决。

### D：Durability（持久性）

**问题**：提交后断电怎么办？

用户转账成功 → 服务器重启 → 钱没了？显然不能接受。

**Commit 之后，必须永久存在。**

---

## 5.4 缓冲池带来的风险

数据在磁盘（慢），MySQL 采用 Buffer Pool（先改内存）。

问题：**内存改完了，磁盘还没写** → 此时断电 → 数据丢失。

怎么办？

---

## 5.5 Redo Log（WAL 思想）

这是 InnoDB 最伟大的设计之一。

### 生活例子

老板让你统计 100 万份订单，你不会直接修改总表，而是先记笔记。笔记非常小，以后电脑坏了还能恢复。

数据库也一样。

### Redo Log 本质

- Redo = **重新执行**
- 记录：**做了什么修改**（不是保存整个数据页）

例如：`Page 8, Offset 100, 写入 200`。类似**操作日志**。

### 为什么快？

修改一条记录：
- 方案 1：直接写数据页 → **16KB**
- 方案 2：写 Redo Log → **几十字节**

显然 Redo Log 更快。

### WAL（Write Ahead Logging）

数据库最核心思想之一：**先写日志，再写数据**。

```text
修改 Buffer Pool → 写 Redo Log → Commit 成功 → 后台慢慢刷盘
```

---

## 5.6 崩溃恢复（Crash Recovery）

崩溃恢复分**两轮**：

### 第一轮：Redo — 重放所有修改

扫描 Redo Log，**不管事务有没有提交**，把所有修改重放一遍。

```text
Redo Log 里记录了什么，就原样执行什么。
```

为什么要重放未提交的事务？因为 Redo Log 遵循 WAL，日志可能已经落盘，但 Commit 标记还没来得及写。所以第一轮先全部重放，确保数据页回到崩溃前的样子。

### 第二轮：Undo — 回滚未提交的事务

第一轮之后，已提交和未提交的修改**都在数据页里了**。现在扫描 Undo Log，找到那些**没有 Commit 标记**的事务：

```text
trx_id=9527: BEGIN, A-100, 没 Commit → 用 Undo 把 A 的值改回 1000
trx_id=9528: BEGIN, B+100, COMMIT → 不需要回滚
```

用 Undo Log 把未提交事务的修改**逐条撤销**。

### 两步之后的结果

```text
已提交的事务 → 数据保留 ✅
未提交的事务 → 全部回滚 ✅
数据回到一致性状态 ✅
```

**总结**：Redo 保证"丢了的不丢"，Undo 保证"不该有的没有"。

---

## 5.7 Undo Log

Redo Log 解决：**已提交的如何恢复**。但**没提交的怎么办**？

例如：`BEGIN; A-100; (系统崩溃)`。恢复后应该看到"没发生转账"，而不是"A 少 100"。

于是 Undo Log 出现。Undo = **撤销**。本质：**保存历史版本**。

- 原始：`balance=1000`
- 修改：`balance=900`
- Undo Log 记录：`旧值=1000`

如果事务失败：读取 Undo Log → 恢复 `balance=1000`。

---

## 5.8 Redo 与 Undo 的关系

```text
                事务
                  │
        ┌─────────┴─────────┐
        │                   │
        ▼                   ▼
   Redo Log            Undo Log
   新状态                旧状态
   用于恢复              用于回滚
```

- **Redo**：面向未来，记录"我做了什么"
- **Undo**：面向过去，记录"我原来是什么"

---

## 5.9 事务的真实结构

```text
Transaction
│
├── Buffer Pool
├── Redo Log
├── Undo Log
├── Lock
└── Commit
```

执行 `BEGIN; UPDATE ...; COMMIT;` 实际上发生：

### 用具体例子走一遍

假设当前数据：

```text
table: account
id=1, balance=1000
```

执行：

```sql
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

每一步发生了什么：

#### ① BEGIN — 事务开始

分配一个事务 ID，比如 `trx_id=9527`。此时什么都没改，只是标记"接下来的操作属于事务 9527"。

#### ② 生成 Undo — 保存旧值

在修改之前，先把**当前值**记到 Undo Log：

```text
Undo Log 写入：
  trx_id=9527
  table: account, id=1
  old_value: balance=1000        ← "我原来是什么"
```

作用：如果事务失败或回滚，用 `old_value=1000` 覆盖回去。

#### ③ 修改 Buffer Pool — 内存中改数据

InnoDB 把数据所在的页（比如 `Page 8`）加载到 Buffer Pool，**在内存中**把 `balance=1000` 改为 `balance=900`：

```text
Buffer Pool（内存）：
  Page 8, Offset 200 → balance = 900   ← 磁盘上还是 1000！
```

此时磁盘不动，只有内存变了。

#### ④ 生成 Redo — 记录"我做了什么"

把这次修改记录到 Redo Log Buffer（也在内存）：

```text
Redo Log Buffer（内存）：
  trx_id=9527
  Page 8, Offset 200
  新值: 900
  "把 Page 8 的第 200 字节改成 900"
```

几十个字节，不是 16KB 整页。

#### ⑤ 写 Redo 磁盘 — Commit 前必须落盘

Commit 之前，先把 Redo Log **刷到磁盘**（这是 WAL 的关键）：

```text
Redo Log File（磁盘）：✅
  trx_id=9527, Page 8, Offset 200, 新值=900
```

此时 Redo 已持久化。断电也不怕——重启后可以重放 Redo 恢复数据。

#### ⑥ Commit — 事务提交

Redo 落盘成功后，标记事务为已提交。**到这一步，用户收到"转账成功"。**

```text
事务状态：COMMITTED ✅
```

后台 InnoDB 会在**空闲时**把 Buffer Pool 的脏页慢慢写回磁盘（Checkpoint），但这和用户无关了——Commit 之后数据就是安全的。

---

### 对比：每一步如果断电会怎样？

| 断电时刻 | 发生了什么 | 恢复结果 |
|----------|-----------|---------|
| ② Undo 之后 | 旧值已记录，新值还没改 | 回滚（Undo 里也没啥好恢复的） |
| ③ 改内存之后 | 内存变了，Redo 还没写 | 内存丢失 → 恢复后 = 旧值 1000 |
| ⑤ Redo 落盘之后 | Redo 已持久化 | **重放 Redo → 恢复 = 新值 900** |
| ⑥ Commit 之后 | 一切正常 | 已提交，数据安全 |
