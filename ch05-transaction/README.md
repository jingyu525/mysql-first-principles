# 第五章：事务与 ACID

> 索引解决「如何快速找到数据」，事务解决「如何保证数据永远正确」。这是从性能问题进入正确性问题的转折点。

---

## 5.1 为什么需要事务？

### 假设：一条 SQL 一定是原子的吗？

```sql
UPDATE account SET balance = balance - 100 WHERE id = 1;
```

很多初学者认为：执行 → 成功。但实际上 MySQL 最终会拆成很多步骤：

```text
找到记录 → 读取 Page → 修改内存 → 更新索引 → 写日志 → 写回磁盘
```

实际上是**多个动作**，不是一个动作。

### 转账案例

```text
A = 1000, B = 1000
A - 100, B + 100
```

逻辑上是"转账"一件事，实际上是两件事。如果 A 扣钱成功 → 机器断电 → A=900, B=1000，**钱消失了**。

### 第一性原理

事务的本质：**让多个操作看起来像一个操作**。

**All or Nothing**：要么全部成功，要么全部失败。

---

## 5.2 事务是什么？

事务可以理解为一个**状态机**：

```text
          Begin
             │
             ▼
       Running
         /   \
        /     \
 Commit       Rollback
```

数据库永远不允许"执行一半结束"这种状态暴露给用户。

---

## 5.3 ACID 逐一推导

很多人背 ACID 然后忘了。实际上 ACID 是四个问题：

### A：Atomicity（原子性）

**问题**：做了一半怎么办？

**解决**：全部成功 or 全部失败。

转账：`A-100` 和 `B+100` 必须一起成功。

### C：Consistency（一致性）

**问题**：数据规则会不会被破坏？

总金额 2000。转账前 `1000+1000`，转账后 `900+1100`，仍然 `2000`。这叫一致性。

### I：Isolation（隔离性）

**问题**：多个事务同时执行怎么办？

事务 A 修改余额，事务 B 查询余额。B 应该看到旧值还是新值？这就是隔离问题，后面 MVCC 专门解决。

### D：Durability（持久性）

**问题**：提交后断电怎么办？

用户转账成功 → 服务器重启 → 钱没了？显然不能接受。

**Commit 之后，必须永久存在。**

---

## 5.4 缓冲池带来的风险

数据在磁盘（慢），MySQL 采用 Buffer Pool（先改内存）。

问题：**内存改完了，磁盘还没写** → 此时断电 → 数据丢失。

怎么办？

---

## 5.5 Redo Log（WAL 思想）

这是 InnoDB 最伟大的设计之一。

### 生活例子

老板让你统计 100 万份订单，你不会直接修改总表，而是先记笔记。笔记非常小，以后电脑坏了还能恢复。

数据库也一样。

### Redo Log 本质

- Redo = **重新执行**
- 记录：**做了什么修改**（不是保存整个数据页）

例如：`Page 8, Offset 100, 写入 200`。类似**操作日志**。

### 为什么快？

修改一条记录：
- 方案 1：直接写数据页 → **16KB**
- 方案 2：写 Redo Log → **几十字节**

显然 Redo Log 更快。

### WAL（Write Ahead Logging）

数据库最核心思想之一：**先写日志，再写数据**。

```text
修改 Buffer Pool → 写 Redo Log → Commit 成功 → 后台慢慢刷盘
```

---

## 5.6 崩溃恢复（Crash Recovery）

事务已提交，Redo Log 已落盘，数据页还在内存 → 突然断电。

数据库启动：扫描 Redo Log → 发现这些修改已提交 → 重新执行 Redo → **恢复成功**。

---

## 5.7 Undo Log

Redo Log 解决：**已提交的如何恢复**。但**没提交的怎么办**？

例如：`BEGIN; A-100; (系统崩溃)`。恢复后应该看到"没发生转账"，而不是"A 少 100"。

于是 Undo Log 出现。Undo = **撤销**。本质：**保存历史版本**。

- 原始：`balance=1000`
- 修改：`balance=900`
- Undo Log 记录：`旧值=1000`

如果事务失败：读取 Undo Log → 恢复 `balance=1000`。

---

## 5.8 Redo 与 Undo 的关系

```text
                事务
                  │
        ┌─────────┴─────────┐
        │                   │
        ▼                   ▼
   Redo Log            Undo Log
   新状态                旧状态
   用于恢复              用于回滚
```

- **Redo**：面向未来，记录"我做了什么"
- **Undo**：面向过去，记录"我原来是什么"

---

## 5.9 事务的真实结构

```text
Transaction
│
├── Buffer Pool
├── Redo Log
├── Undo Log
├── Lock
└── Commit
```

执行 `BEGIN; UPDATE ...; COMMIT;` 实际上发生：

```text
BEGIN
 ↓
生成 Undo
 ↓
修改 Buffer Pool
 ↓
生成 Redo
 ↓
写 Redo 磁盘
 ↓
Commit
```

