# 第三章：存储引擎

> 真正的主线是：磁盘 → Page（页）→ Buffer Pool（缓存）→ B+Tree → 索引。事务、MVCC、锁都是建立在数据已经存储起来的基础之上。

---

## 3.1 从磁盘开始

### 隐含假设

很多人以为 MySQL 直接管理数据。实际上：

```text
MySQL → 操作系统 → 文件系统 → 磁盘
```

`INSERT INTO user VALUES(1,'张三')` 最终一定会变成：**往磁盘文件写字节**。

**数据库本质上是在解决：如何高效读写磁盘。**

---

## 3.2 为什么磁盘慢？

先建立量级概念：

| 操作 | 相对速度 |
|------|---------|
| CPU 访问寄存器 | ≈ 1 |
| 访问内存 | ≈ 10 |
| SSD | ≈ 10000+ |

数据库性能瓶颈通常不是 CPU，而是**磁盘 IO**。

**数据库设计第一原则：减少磁盘访问次数。**

---

## 3.3 Page（页）—— 数据库最小读写单位

### 隐含假设

数据库会不会这样读数据？"读取 1 条记录" → **不会**。

磁盘读取最小单位是 **Block（块）**，数据库对应概念是 **Page（页）**。

InnoDB 默认：**16KB** 一个 Page。

```text
Page = 数据库最小读写单位
```

### 举例

表 user 有 100 条记录，正好放进一个 Page。

```sql
SELECT * FROM user WHERE id=1;
```

虽然只要 1 条记录，但实际上：**整个 Page（16KB）全部加载**。

**第一个核心认知：数据库以 Page 为单位工作，不是以行工作。**

---

## 3.4 为什么需要 Buffer Pool？

如果每次查询都是：磁盘 → Page → 返回结果。`SELECT * FROM user WHERE id=1` 一天 100 万次 → 难道磁盘读 100 万次？显然不合理。

### 第一性原理

Buffer Pool = **缓存**。本质：**用内存换性能**。

```text
            MySQL
               │
               ▼
      Buffer Pool
               │
       命中(Hit) → 返回结果
               │
       未命中(Miss) → 磁盘
```

类似 Go 里的 `sync.Pool` 思想：先看缓存有没有，没有再创建。

### 工作流程

第一次查询 → Page 不存在 → 磁盘读取 Page → 放入 Buffer Pool → 返回数据。

第二次查询（恰好同一 Page）→ 直接内存读取 → 无需磁盘 → 速度提升几千倍。

---

## 3.5 Page 为什么是 16KB？

| Page 大小 | 问题 |
|-----------|------|
| 太小（如 512B）| 树高度增加，IO 次数增加 |
| 太大（如 1MB）| 内存浪费，缓存效率下降 |

MySQL 选择 **16KB** 作为平衡点。

---

## 3.6 Page 内部结构

```text
+----------------+
| Header         |  ← 页号、页类型、事务信息
+----------------+
| Record         |
| Record         |
| Record         |
+----------------+
| Directory      |  ← 数据位置，方便快速定位
+----------------+
```

Page 类似一个**小型数据库**。

---

## 3.7 Page 与 B+Tree 的关系

**为什么数据库选择 B+Tree？因为 Page 存在。**

普通二叉树每个节点很小 → 一次 IO 只读取一个节点 → 浪费。

数据库设计：**一个节点 = 一个 Page（16KB）**。里面能放数百甚至上千个 Key。

```text
                [Page]
       100 200 300 400 500
         /    |    |    \
```

### 实际高度

一个 Page 存 1000 个 Key：

```text
三层：1000 × 1000 × 1000 = 10 亿
```

**查 10 亿数据 ≈ 3 次 IO。** 这就是 B+Tree 恐怖的地方。

---

## 3.8 聚簇索引

叶子节点存什么？

- 方案 1：叶子节点存地址
- 方案 2：叶子节点存完整数据 ✓

InnoDB 选择**完整数据**。即：**主键索引 = 数据本身**。

称为 **Clustered Index（聚簇索引）**。

```text
Root → Middle → Leaf
Leaf: (id, name, age)  ← 完整数据
```

`SELECT * FROM user WHERE id=1` → 只查一次树，直接得到结果。

---

## 3.9 二级索引 & 回表

查询 `WHERE name='张三'`，name 索引叶子节点存的是**主键 ID**，不是完整数据。

```text
name索引：张三 → id=1
         ↓
主键索引：id=1 → 完整数据
```

需要两棵树 → 这就是**回表**。

---

## 当前认知地图

```text
磁盘
  ↓
Page (16KB)
  ↓
Buffer Pool
  ↓
B+Tree
  ↓
聚簇索引
  ↓
二级索引
  ↓
回表
```
