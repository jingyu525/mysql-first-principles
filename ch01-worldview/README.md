# 第一章：数据库世界观

> 如果你是一个 MySQL 初学者，不要从 SQL 语法开始学，而是先建立一套「数据库世界观」。

---

## 1.1 为什么会有 MySQL？

### 第一性原理

数据库本质上是在解决三个问题：

#### 问题1：数据存哪里？

假设你开发一个微信，用户注册：张三, 18岁, 北京。

最简单的方法：放在内存里。

```go
user := User{
    Name: "张三",
    Age: 18,
}
```

但是程序重启 → **数据没了**。于是需要**持久化存储**：写到磁盘。

---

#### 问题2：如何快速找到数据？

假设有 1000 万用户，你要找"张三"。

最原始方式：第1个 → 第2个 → 第3个 → ... → 第1000万个。

这叫 **线性查找，O(n)**，太慢。

于是需要 **索引（Index）**，让查找变成 **O(logN)** 甚至 **O(1)**。

---

#### 问题3：多人同时修改怎么办？

账户余额 100 元，同时两个人扣款：`-30` 和 `-50`。

如果处理不好，余额可能变成 70 或 50，结果错误。

于是需要 **事务（Transaction）**：保证要么全部成功，要么全部失败。

---

## 1.2 数据库是什么？

很多初学者的隐含假设：**数据库 = 一个软件**。这是错误的。

更准确地说：

```text
数据库 = 数据管理系统（DBMS: Database Management System）
```

例如：MySQL、PostgreSQL、Oracle、SQL Server。

本质：**帮助你管理磁盘上的数据**。

---

## 1.3 MySQL 内部架构

```text
             应用程序
                 |
              SQL语句
                 |
                 v
+--------------------------------+
|         MySQL Server           |
+--------------------------------+
| Parser      SQL解析器          |
| Optimizer   优化器            |
| Executor    执行器            |
+--------------------------------+
                 |
                 v
+--------------------------------+
|       Storage Engine           |
|        (InnoDB)                |
+--------------------------------+
                 |
                 v
             磁盘文件
```

### 隐含知识

`SELECT * FROM user` 并不是 SQL 直接读取数据。

实际路径：

```text
SQL → Parser → Optimizer → Executor → InnoDB → 磁盘
```

## 1.4 表（Table）是什么？

### 第一性原理

表其实就是**结构化数据集合**。

| id | name | age |
| -- | ---- | --- |
| 1  | 张三   | 18  |
| 2  | 李四   | 20  |

数学上：**表 = 二维数组** 或 **表 = 行集合**。

### 隐含知识

为什么需要表？因为计算机擅长处理**规则结构**，不擅长处理**任意文本**。所以必须设计**列（Column）**。

---

## 1.5 主键是什么？

假设："张三" "张三" "张三" — 怎么区分？

所以需要 `id`：

```text
1 张三
2 张三
3 张三
```

主键本质：**唯一标识符**，像身份证。

### 隐含知识

为什么每张表都建议有主键？因为**计算机识别的是 ID，不是名字**。

```sql
WHERE id = 100    -- 远快于
WHERE name='张三'
```

---

## 1.6 索引是什么？

这是 MySQL 最核心知识之一。

### 第一性原理

查字典：不会从第一页开始翻，而是查**目录**。目录就是索引。

- 没有索引：**全表扫描**
- 有索引：**直接定位**

### 隐含知识

索引不是数据。索引是**帮助找到数据的数据**，即**元数据（Meta Data）**。

---

## 1.7 B+树是什么？

MySQL 默认索引：**B+Tree**。

为什么不是数组？因为**磁盘很慢**。真正耗时的是**磁盘 IO**，不是 CPU 计算。

所以设计目标：**减少磁盘读取次数**。

简化理解：

```text
             [50]
          /        \
       <50        >50

                 [50]
             /         \
          [20]       [80]
         /   \       /   \
      ...   ...   ...   ...
```

优势：**查找快、范围查询快、排序快**。

---

## 1.8 事务是什么？

### 第一性原理

事务 = **一组操作**。

转账：`A -100`、`B +100`。必须同时成功。

如果 A 扣了、B 没加 → 数据错了。

事务保证 **原子性**。

### ACID

事务四大特性：

- **A**tomicity：要么全做，要么不做
- **C**onsistency：数据规则不被破坏
- **I**solation：多个事务互不影响
- **D**urability：写入后断电不丢

---

## 1.9 MVCC 是什么？

很多人直接背概念。其实本质是：**提高并发能力**。

问题：1000 人同时查询怎么办？如果加锁 → 所有人排队 → 很慢。

于是：**多版本并发控制（MVCC）**。

思想：**给数据保留多个历史版本**。读旧版本，写新版本，互不阻塞。

---

## 1.10 学习路线图

```text
第一阶段：数据库世界观
    → 数据库是什么、表、行/列、主键、外键

第二阶段：SQL 语言
    → SELECT、INSERT、UPDATE、DELETE

第三阶段：多表关系
    → 一对一、一对多、多对多、JOIN

第四阶段：索引
    → B+Tree、聚簇索引、二级索引、覆盖索引、最左匹配

第五阶段：事务
    → ACID、锁、MVCC、隔离级别

第六阶段：MySQL 内核
    → Parser、Optimizer、Executor、InnoDB

第七阶段：性能优化
    → EXPLAIN、慢查询、索引优化、SQL 优化
```

对于有系统编程背景的读者，推荐学习顺序：

```text
数据结构 → B+Tree → 索引 → 事务 → MVCC → 锁 → EXPLAIN → InnoDB 源码思想
```

因为 MySQL 本质上也是一个大型系统软件。你在学 Go Runtime 时接触的：内存管理、调度器、状态机、锁、并发控制 — 这些思想在 MySQL 内核里几乎都会再次出现，只不过对象从「goroutine」变成了「数据和事务」。
