# 第二章：从数据结构到 B+树

> 很多教程的问题在于：直接讲索引、直接讲事务、直接讲 MVCC。但它们都建立在一堆没有说出来的假设上。下面我们从第一性原理开始，把每一层隐含假设全部展开。

---

## 2.1 数组与链表

先不要想 MySQL。先想一个问题：**计算机如何保存数据？**

假设：张三、李四、王五、赵六，放进内存。

### 数组

```text
[张三][李四][王五][赵六]
```

特点：**查找快、插入慢、删除慢**。（内存连续）

### 链表

```text
张三 → 李四 → 王五 → 赵六
```

特点：**插入快、删除快、查找慢**。

### 隐含知识

数据库的本质问题：**如何在查找、插入、删除之间平衡？**

---

## 2.2 二分查找 & 二叉搜索树（BST）

假设 1000 万条数据，找 `id=8888888`。顺序找 → O(n)，太慢。

于是想到**二分查找**：先看中间，比目标大向左，比目标小向右。复杂度 O(logN)，非常快。

但有个隐藏假设：**数据必须有序**。

于是出现 **二叉搜索树（BST）**：

```text
          50
        /    \
      30      80
```

查找 O(logN)，很好。

---

## 2.3 为什么 BST 还不够？→ 平衡树

如果插入顺序是 1, 2, 3, 4, 5, 6...

```text
1
 \
  2
   \
    3
     \
      4
```

退化成**链表**，复杂度 O(n)，崩了。

于是出现**平衡树**（AVL Tree、Red Black Tree）。

隐含假设：**树必须保持平衡**，否则查找性能不稳定。

---

## 2.4 为什么还要 B 树？→ 磁盘 IO 瓶颈

到这里大部分教程停了，但 MySQL 还没出现。

关键问题：**数据不在内存，而在磁盘**。

### 速度差

- CPU 访问：**纳秒级**
- 磁盘：**毫秒级**

**差距 10 万倍以上。**

于是问题变成：**如何减少磁盘读取次数**，而不是如何减少 CPU 计算次数。

---

## 2.5 B 树出现

BST 一个节点两个孩子，高度很高。1000 万数据可能 **20 多层** → 20 次磁盘 IO。

于是：一个节点存更多数据：

```text
         [30]
      /        \
 [10 20]    [40 50]
```

高度变矮。从 20 层变成 3 层。磁盘 IO 骤降。**B Tree 诞生**。

---

## 2.6 为什么是 B+Tree？

MySQL 真正使用的是 **B+Tree**，不是 B Tree。

数据库最常见操作：范围查询。

```sql
SELECT * FROM user WHERE id BETWEEN 100 AND 1000;
```

- B Tree：数据分散，范围查询来回跳，效率一般
- B+Tree：叶子节点形成链表 `1 → 2 → 3 → 4 → 5`，范围查询顺着扫即可

所以 B+Tree 同时满足：**查找快、排序快、范围查询快**。

---

## 2.7 索引出现

索引本质：**帮助找到数据的数据结构**。

类似电话簿：张三 → 第 100 页，李四 → 第 250 页。

```sql
CREATE INDEX idx_name ON user(name);
```

实际上：**建立一棵 B+Tree**。

---

## 2.8 为什么事务出现？

索引解决**查找问题**，但没解决**正确性问题**。

银行转账：A=100, B=100。执行 A-=50, B+=50。

如果中间断电：A=50, B=100 → **钱没了**。

于是**事务（Transaction）**出现。

事务本质：**状态机**。只有两种状态：**Commit** 或 **Rollback**。

---

## 2.9 为什么 MVCC 出现？

事务解决**一致性**。但出现新问题：**并发**。

假设 10000 人在查数据，1 人在改数据。最简单办法：加锁 → 所有查询等待 → 性能崩。

新思想：**不要让读写互相阻塞**。

实现：保存历史版本（Version1, Version2, Version3）。读旧版本，写新版本。

这就是 **MVCC（Multi-Version Concurrency Control）**。

第一性原理：**空间换时间**。

---

## 2.10 为什么还需要锁？

MVCC 只能解决**读写冲突**，不能解决**写写冲突**。

余额 100，事务 A 减 50，事务 B 减 80。同时执行 → 必须决定谁先改。

所以锁仍然存在：Record Lock、Gap Lock、Next-Key Lock。

---

## 2.11 执行计划（EXPLAIN）

为什么有的 SQL 1ms，有的 10 秒？因为数据库会选择执行路径。

优化器负责决定：走索引还是全表扫描？

EXPLAIN 作用：**观察优化器决策**。类似 Go 里的 `go tool pprof`。

---

## 2.12 InnoDB 源码思想

InnoDB 其实是**一个管理数据的操作系统**。

内部组件：

| 组件 | 类比 |
|------|------|
| Buffer Pool | 内存管理 |
| Page | 虚拟内存页 |
| Redo Log | WAL |
| Undo Log | 历史快照 |
| Lock Manager | 同步机制 |
| Transaction Manager | 状态机 |
| B+Tree | 文件系统目录 |

---

## 完整推导链路

```text
数据结构 → 树 → B树 → B+树 → 索引 → 事务 → MVCC → 锁 → 优化器 → InnoDB
```

### Go Runtime ↔ InnoDB 对照表

| Go Runtime | InnoDB |
|-----------|--------|
| G | Transaction |
| GMP 调度 | Lock/MVCC 协调 |
| Heap | Buffer Pool |
| GC | Purge Thread |
| pprof | EXPLAIN |
| Channel | Lock Wait Queue |
| 状态机 | Transaction State |
