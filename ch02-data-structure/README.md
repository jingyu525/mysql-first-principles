# 第二章：从数据结构到 B+树

> 很多教程的问题在于：直接讲索引、直接讲事务、直接讲 MVCC。但它们都建立在一堆没有说出来的假设上。下面我们从第一性原理开始，把每一层隐含假设全部展开。

---

## 2.1 数组与链表

先不要想 MySQL。先想一个问题：**计算机如何保存数据？**

假设：张三、李四、王五、赵六，放进内存。

### 数组

```text
[张三][李四][王五][赵六]
```

特点：**查找快、插入慢、删除慢**。（内存连续）

### 链表

```text
张三 → 李四 → 王五 → 赵六
```

特点：**插入快、删除快、查找慢**。

### 隐含知识

数据库的本质问题：**如何在查找、插入、删除之间平衡？**

---

## 2.2 二分查找 & 二叉搜索树（BST）

假设 1000 万条数据，找 `id=8888888`。顺序找 → O(n)，太慢。

于是想到**二分查找**：先看中间，比目标大向左，比目标小向右。复杂度 O(logN)，非常快。

但有个隐藏假设：**数据必须有序**。

于是出现 **二叉搜索树（BST）**：

```text
          50
        /    \
      30      80
```

查找 O(logN)，很好。

---

## 2.3 为什么 BST 还不够？→ 平衡树

如果插入顺序是 1, 2, 3, 4, 5, 6...

```text
1
 \
  2
   \
    3
     \
      4
```

退化成**链表**，复杂度 O(n)，崩了。

于是出现**平衡树**（AVL Tree、Red Black Tree）。

隐含假设：**树必须保持平衡**，否则查找性能不稳定。

---

## 2.4 为什么还要 B 树？→ 磁盘 IO 瓶颈

到这里大部分教程停了，但 MySQL 还没出现。

关键问题：**数据不在内存，而在磁盘**。

### 速度差

- CPU 访问：**纳秒级**
- 磁盘：**毫秒级**

**差距 10 万倍以上。**

于是问题变成：**如何减少磁盘读取次数**，而不是如何减少 CPU 计算次数。

---

## 2.5 B 树出现

BST 一个节点两个孩子，高度很高。1000 万数据可能 **20 多层** → 20 次磁盘 IO。

思路很简单：**让每个节点多存几个值，多分几个叉**，树自然就矮了。

B 树结构：**每个节点存多个 key，同时也存 key 对应的完整数据（data）**。

```text
                    [30|data]
                   /          \
          [10|data,20|data]    [40|data,50|data]
         /      |       \       /      |       \
      [...]   [...]    [...]  [...]   [...]    [...]
```

高度从 20 层变成 3 层，磁盘 IO 骤降。**B Tree 诞生**。

**B 树的特点**：每个节点（无论是内部节点还是叶子节点）都同时存储 key + data。

---

## 2.6 为什么是 B+Tree？

MySQL 真正使用的是 **B+Tree**，不是 B Tree。

### B Tree 的两个问题

**问题一：范围查询低效**

数据库最常见操作：

```sql
SELECT * FROM user WHERE id BETWEEN 100 AND 1000;
```

B 树的数据分散在所有节点里。遍历 100→1000 需要在树的不同层级之间来回跳，**中序遍历跨越多个节点**，效率一般。

**问题二：每个节点存了 data，能装的 key 就少了**

B 树每一个 key 旁边都带着完整的 data。一个 Page 16KB，data 越大，一个节点能装的 key 越少 → **树更高 → IO 次数更多**。

---

### B+Tree 做了什么？

B+Tree 做了两个关键改造：

**改造一：data 只放在叶子节点**

内部节点只存 key 做导航，不存 data。这样一来，一个内部节点可以装大量 key，树更矮。

**改造二：叶子节点之间用链表串联**

所有叶子节点形成一个**有序双向链表**。

```text
               B Tree                               B+Tree

          [30|data]                           [30]  ← 只存key，无data
         /          \                        /        \
  [10|d,20|d]   [40|d,50|d]          [10,20]          [40,50]  ← 只存key
   /   |   \     /   |   \            /     \           /     \
 ...  ...  ...  ...  ...  ...    [1|d→2|d→3|d]  [100|d→101|d→...]
                                       ↕                  ↕
                                    ← 双向链表 →
```

---

### 一张表看懂区别

| | B Tree | B+Tree |
|---|---|---|
| data 存在哪 | **所有节点**都存 data | **只有叶子节点**存 data |
| 内部节点作用 | 导航 + 存数据 | 只做导航（存更多 key） |
| 叶子节点之间 | **没有连接** | **双向链表**串联 |
| 等值查找 | O(logN) | O(logN)，差不多 |
| 范围查询 | 中序遍历，来回跳 | **顺着链表扫**，连续 IO |
| 全表扫描 | 中序遍历整棵树 | **扫一遍叶子链表**即可 |
| 树高（同样数据量） | 更高 | **更矮**，IO 更少 |

---

### 为什么 B+Tree 更适合数据库？

三个字：**磁盘友好**。

1. **范围查询快** — 找到起点，顺着叶子链表扫就行，不用在层级之间跳来跳去
2. **IO 次数少** — 内部节点只存 key，一个 Page 能装更多 key → 树更矮 → 查一次要读的 Page 更少
3. **全表扫描高效** — 不需要遍历整棵树，只要扫一遍叶子节点的链表即可（这个很重要，有时优化器真的会选全表扫描）

所以 B+Tree 同时满足：**查找快、排序快、范围查询快**。

---

### B+Tree 必须排序 —— 这是它的"基因"

你可能在第四章会看到一句话：「因为 B+Tree 必须排序，所以每张表要有主键」。为什么 B+Tree **天生必须有序**？

拆开看，B+Tree 有两层查找：

```text
第一层：节点内查找 → 二分查找（必须有序！）
第二层：节点间路由 → 大小比较决定往哪个子节点走（必须有序！）
```

**节点内**：一个节点存了几十个 key，怎么快速确定目标 key 在不在这个节点里？二分查找。二分查找的前提就是 key 从小到大排好序。如果是乱的，只能挨个找，跟链表没区别。

**节点间**：B+Tree 的路由规则是：

```text
左边子节点所有 key < 当前最小 key < 右边子节点所有 key
```

```text
              [20, 30]
             /    |    \
  [5,8,10]   ← <20  →   [23,25]   ← 20~30  →   [33,40]  ← >30
```

一旦 key 不是有序的，这个"按大小分流"的路由逻辑就崩了。你不知道该往哪个子节点走。

**一句话：有序是 B+Tree 能 O(logN) 查找的前提，去掉有序，B+Tree 不是 B+Tree。**

这也是为什么后续章节你会反复看到：主键是排序依据、联合索引有最左匹配原则、`ORDER BY` 可以利用索引 —— 它们都建立在这个基础上：**B+Tree 里面的数据是有序的**。

---

## 2.7 索引出现

索引本质：**帮助找到数据的数据结构**。

类似电话簿：张三 → 第 100 页，李四 → 第 250 页。

```sql
CREATE INDEX idx_name ON user(name);
```

实际上：**建立一棵 B+Tree**。

---

## 2.8 为什么事务出现？

索引解决**查找问题**，但没解决**正确性问题**。

银行转账：A=100, B=100。执行 A-=50, B+=50。

如果中间断电：A=50, B=100 → **钱没了**。

于是**事务（Transaction）**出现。

事务本质：**状态机**。只有两种状态：**Commit** 或 **Rollback**。

---

## 2.9 为什么 MVCC 出现？

事务解决**一致性**。但出现新问题：**并发**。

假设 10000 人在查数据，1 人在改数据。最简单办法：加锁 → 所有查询等待 → 性能崩。

新思想：**不要让读写互相阻塞**。

实现：保存历史版本（Version1, Version2, Version3）。读旧版本，写新版本。

这就是 **MVCC（Multi-Version Concurrency Control）**。

第一性原理：**空间换时间**。

---

## 2.10 为什么还需要锁？

MVCC 只能解决**读写冲突**，不能解决**写写冲突**。

余额 100，事务 A 减 50，事务 B 减 80。同时执行 → 必须决定谁先改。

所以锁仍然存在：Record Lock、Gap Lock、Next-Key Lock。

---

## 2.11 执行计划（EXPLAIN）

为什么有的 SQL 1ms，有的 10 秒？因为数据库会选择执行路径。

优化器负责决定：走索引还是全表扫描？

EXPLAIN 作用：**观察优化器决策**。类似 Go 里的 `go tool pprof`。

---

## 2.12 InnoDB 源码思想

InnoDB 其实是**一个管理数据的操作系统**。

内部组件：

| 组件 | 类比 |
|------|------|
| Buffer Pool | 内存管理 |
| Page | 虚拟内存页 |
| Redo Log | WAL |
| Undo Log | 历史快照 |
| Lock Manager | 同步机制 |
| Transaction Manager | 状态机 |
| B+Tree | 文件系统目录 |

---

```text
数据结构 → 树 → B树 → B+树 → 索引 → 事务 → MVCC → 锁 → 优化器 → InnoDB
```
