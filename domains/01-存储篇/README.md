# 领域视角：存储篇

> 数据最终存在磁盘上。磁盘如何被组织？如何被高效读写？这是整个 InnoDB 的物理基础。

---

## 核心问题

| 问题 | 答案 |
|------|------|
| 数据存在哪？ | 磁盘文件，MySQL → 操作系统 → 文件系统 → 磁盘 |
| 为什么磁盘慢？ | SSD 比内存慢 10000+ 倍 |
| 如何高效读写？ | 减少磁盘访问次数 |
| 最小读写单位是什么？ | Page（16KB） |
| 数据如何组织？ | B+Tree → 聚簇索引 / 二级索引 |

---

## 四大主题

### Page

数据库不会"读一条记录"，而是读一个 Page（16KB）。**Page 是最小 IO 单位**。

→ 详见：[第三章：存储引擎 §3.3-3.7](../../ch03-storage-engine/README.md)

### B+Tree

为什么是 B+Tree 而不是 BST？因为磁盘慢，要尽量减少 IO。B+Tree 一个节点一个 Page，高度极低（10 亿数据 ≈ 3 次 IO）。

→ 详见：[第二章：从数据结构到 B+树](../../ch02-data-structure/README.md) — 完整推导：数组→链表→BST→B树→B+树

### 聚簇索引

InnoDB 中，**表本身就是一棵 B+Tree**。叶子节点存完整数据，主键就是排序依据。

→ 详见：[第四章：索引体系 §4.2](../../ch04-index/README.md)

### 二级索引

非主键索引，叶子节点存主键值（不是完整数据）。查询需要两次查找（回表）。

→ 详见：[第四章：索引体系 §4.3-4.5](../../ch04-index/README.md)

## 知识地图

```text
存储篇
│
├── Page（最小 IO 单位，16KB）
│   ├── Page 内部结构（Header/Record/Directory）
│   ├── Page 与磁盘 Block 的关系
│   └── Page 是 B+Tree 节点、Buffer Pool 缓存单位
│
├── B+Tree（索引数据结构）
│   ├── 为什么不是数组/链表？（O(n) 太慢）
│   ├── 为什么不是 BST？（高度太高，IO 太多）
│   ├── 为什么是 B+Tree？（矮、宽、叶子链表）
│   └── 10 亿数据 ≈ 3 次 IO
│
├── 聚簇索引（表 = B+Tree）
│   ├── 叶子节点存完整数据
│   ├── 主键即排序依据
│   └── 为什么必须有主键？
│
└── 二级索引（非主键索引）
    ├── 叶子节点存主键值
    ├── 回表：二级索引 → 聚簇索引（两次查找）
    ├── 覆盖索引：不回表（性能极高）
    ├── 联合索引：多列排序 + 最左匹配原则
    └── 索引失效：函数运算、前缀模糊、数据分布
```

## 核心推论

1. **索引本质是时间和空间的权衡** — 用额外空间（建 B+Tree）换更少的磁盘 IO
2. **磁盘 IO 是唯一瓶颈** — 所有存储设计都围绕"少读 Page"展开
3. **覆盖索引是最强优化** — 一次索引查询直接返回，避免回表
