## 贯穿始终的设计原则

### 原则一：磁盘 IO 是唯一瓶颈

数据库一切设计的出发点：**CPU 纳秒级，磁盘毫秒级，差距 10 万倍以上。**

| 设计 | 如何体现 |
|------|---------|
| B+Tree | 靠宽节点降低树高 → 减少 IO 次数 |
| Page（16KB） | 一次 IO 读写一个 Page，不是一条记录 |
| Buffer Pool | 缓存热点 Page → 避免重复 IO |
| Redo Log | 写几十字节日志替代 16KB Page 刷盘 |
| 覆盖索引 | 一次查询不回表 → 减少 IO |
| 延迟刷盘 | 批量写入避免频繁随机 IO |

**推论：优化数据库的本质，就是减少 Page 的读写次数。**

---

### 原则二：空间换时间

用额外的存储空间换取更快的访问速度。

| 设计 | 换取了什么 |
|------|-----------|
| 索引（B+Tree） | 用索引占用的磁盘空间换 O(logN) 查找 |
| 覆盖索引 | 用索引中多存的列换不回表 |
| MVCC | 用 Undo Log 保留的历史版本换读写不阻塞 |
| Buffer Pool | 用内存空间换减少磁盘 IO |
| Redo Log | 用额外的日志文件换快速提交 + 崩溃恢复 |

---

### 原则三：Redo/Undo 双日志架构

几乎所有现代数据库都用这种模式：

```text
Redo Log  → 记录"做了什么" → 保证已提交不丢
Undo Log  → 记录"原来是什么" → 保证可回滚 + 提供历史版本
```

这种架构的巧妙之处：
- **Redo 先落盘，数据后落盘** — 提交速度快，不怕崩溃
- **Undo 一石二鸟** — 既服务事务回滚，又为 MVCC 提供版本链
- **两者分离** — Redo 追求写入速度（顺序写），Undo 追求随机读取（版本回退）

### 原则四：分层解耦

```text
MySQL Server（SQL 层）
  ├── Parser     ← 解析 SQL
  ├── Optimizer  ← 选择执行计划
  └── Executor   ← 驱动引擎执行
        │
        ▼
Storage Engine（存储引擎层）  ← 可替换
  └── InnoDB
        │
        ▼
Operating System（文件系统）
```

**分层的好处：**
- SQL 层不关心数据怎么存储（可以用 InnoDB，也可以换 MyISAM、RocksDB）
- 存储引擎层不关心 SQL 语法（只接收读/写 Page 的指令）
- 每层可以独立优化

---

### 原则五：乐观与悲观的混合策略

```text
乐观策略（假设冲突少，事后检查）：
  → MVCC：读不加锁，先读历史版本
  → 乐观锁：CAS 思想，修改时检查版本号

悲观策略（假设冲突多，事先加锁）：
  → Record Lock：修改前先锁记录
  → Gap Lock：锁间隙防止插入
  → Next-Key Lock：默认锁策略
```

**InnoDB 的混合策略：**
- 对于读操作 → 乐观（MVCC，不加锁读旧版本）
- 对于写操作 → 悲观（加锁，保证写写不冲突）
- 这种混合使 InnoDB 在读多写少的场景（Web 应用典型场景）表现极佳

---

### 原则六：延迟与批处理

```text
不立即做的事：
  - 修改 Buffer Pool 后不立即刷盘 → 延迟 Flush
  - Redo Log 不立即应用到数据页 → 延迟应用
  - 删除数据不立即回收空间 → 标记删除，Purge Thread 异步清理

为什么延迟？
  → 合并多次小 IO 为一次大 IO
  → 将随机 IO 转化为顺序 IO
  → 后台线程处理，不阻塞前台请求
```

## 数据库设计的永恒权衡

| 权衡 | 左 | 右 |
|------|------|------|
| 读 vs 写 | 建索引加速读 | 维护索引拖慢写 |
| 速度 vs 一致性 | 延迟刷盘（快但可能丢） | 同步刷盘（慢但安全） |
| 并发 vs 隔离 | MVCC（高并发，读旧版本） | Serializable（低并发，完全隔离） |
| 内存 vs 磁盘 | Buffer Pool 越大越快 | 内存有限，需要淘汰策略 |
| 空间 vs 时间 | MVCC 保留历史版本 | Purge 清理回收空间 |

## 设计哲学总结

```text
数据库不是 SQL 语言。

数据库本质上是一个
专门解决「海量数据的高效存储、检索、一致性、并发控制」
的分布式/集中式操作系统内核。

所有设计的终极目标：
  在保证数据正确的前提下，尽可能地减少磁盘 IO。
```
